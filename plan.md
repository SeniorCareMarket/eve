# Implementation Plan: Columnar Data Structures for Eve

## Overview

Four work items, building bottom-up:

0. **Extend `eve-slab-deftype`** to emit cross-platform code (CLJS + JVM + future targets)
1. **`eve.dataset`** — tabular data: named columns (EveArrays) with atom-native storage
2. **`eve.tensor`** — N-dimensional views over EveArrays (shape/strides, zero-copy)
3. **`eve.dataset.functional` + `eve.dataset.argops`** — column operations

All defined via the extended `eve-slab-deftype`. One type definition, all
platforms. Native column format = EveArray (tmdjs-style typed buffers).
Arrow compatibility is a future foreign-format adapter, not the default.

---

## Step 0: Extend eve-slab-deftype for Cross-Platform

### Problem

`eve-slab-deftype` currently emits CLJS-only code (DataView, Atomics, etc.).
JVM types (JvmEveArray, JvmObj, JvmSlabCtx) are hand-written separately.
This forces type proliferation: `EveArray` + `JvmEveArray`, `Obj` + `JvmObj`.

### Solution

Make the macro platform-aware: check `&env` at expansion time.

```clojure
(defmacro eve-slab-deftype [type-name fields & body]
  (if (:ns &env)
    (emit-cljs-deftype type-name fields body)   ;; existing code path
    (emit-clj-deftype type-name fields body)))   ;; new code path
```

**CLJS path** (existing): `(deftype Foo [offset__] ...)` — resolves via
`resolve-dv!`, reads via `DataView`, Atomics for volatile fields.

**JVM path** (new): `(deftype Foo [^long slab-off sio] ...)` — reads via
`ISlabIO` methods (`-sio-read-i32`, `-sio-read-u8`, etc.), writes via
`-sio-write-i32!` etc.

JVM field read for `^:int32 count` at offset 4:
```clojure
(alloc/-sio-read-i32 sio slab-off 4)
```

JVM field write:
```clojure
(alloc/-sio-write-i32! sio slab-off 4 value)
```

JVM volatile field read (atomic):
```clojure
;; ISlabIO already provides atomic i32 access on JVM via IMemRegion CAS
(alloc/-sio-read-i32-volatile sio slab-off 4)
```

### JVM constructor

```clojure
;; Generated by macro (JVM path)
(defn ->Foo [sio count ...]
  (let [slab-off (alloc/jvm-alloc-offset sio total-size)]
    (alloc/-sio-write-u8! sio slab-off 0 type-id)
    (alloc/-sio-write-i32! sio slab-off 4 count)
    ...
    (Foo. slab-off sio)))
```

### JVM protocol boilerplate

The macro generates JVM implementations of:
- `clojure.lang.ILookup` (field access by keyword)
- `clojure.lang.IHashEq` (hash by slab-off)
- `clojure.lang.Seqable` (seq of field entries)
- `Object.equals`, `Object.toString`

Same pattern as hand-written `JvmObj` and `JvmEveArray`, but auto-generated.

## Step 1: eve.dataset

### Native column format

**Columns ARE EveArrays.** No new column type. EveArray (type-id `0x1D`)
already provides:
- Typed buffers: `:int8` through `:float64`
- Slab-backed: lives in slab classes 0–6
- Cross-platform: JvmEveArray + EveArray from one definition (after Step 0)
- Atomics, SIMD, reduce, map

This is the tmdjs approach: columns are typed arrays, period.

### Dataset type

Defined via the extended `eve-slab-deftype`:

```clojure
(eve-slab-deftype DatasetManifest
  [^:int32 row-count       ;; number of rows
   ^:int32 ncols           ;; number of columns
   ^:int32 col-list-off])  ;; slab offset of column descriptor block
```

The column descriptor block is a separate slab allocation (variable-length,
like Obj's schema encoding). Format:

```
Per column (repeated ncols times):
[name-len : u8]         column name length
[name : name-len bytes] UTF-8 column name
[dtype : u8]            subtype code (reuse array subtype codes)
[col-slab-off : u32]    slab-qualified offset of EveArray block
```

This pattern matches how `eve.obj` encodes its schema: a fixed-size deftype
header references a variable-length descriptor block.

### Wrapper type

On top of the raw slab deftype, a lightweight wrapper provides the user API:

```clojure
;; .cljc — works on all platforms
(defn dataset [col-map]
  ;; col-map: {keyword → EveArray}
  ;; 1. Validate all columns have same length
  ;; 2. Allocate column descriptor block
  ;; 3. Construct DatasetManifest via ->DatasetManifest
  ;; 4. Return wrapper with parsed column map
  ...)
```

### Core API

```clojure
;; Construction
(dataset {:price  (arr/eve-array :float64 [10.5 20.3])
          :qty    (arr/eve-array :int32 [100 200])})

;; Column access (zero-copy — returns the EveArray directly)
(column ds :price)          ;; → EveArray
(column-names ds)           ;; → [:price :qty]
(row-count ds)              ;; → 2
(dtypes ds)                 ;; → {:price :float64 :qty :int32}

;; Structural ops (new Dataset, structural sharing on unchanged columns)
(select-columns ds [:price])
(add-column ds :total some-array)
(drop-column ds :qty)
(rename-columns ds {:price :cost})

;; Row ops (new Dataset with new EveArray blocks)
(filter-rows ds :price #(> % 15.0))
(sort-by-column ds :price :desc)
(head ds 10)
(tail ds 10)
(slice ds 5 15)             ;; zero-copy sub-view
```

### Slab serialization (atom-storable)

New type-id: `EVE_DATASET_SLAB_TYPE_ID = 0x1F`

The DatasetManifest slab block IS the serialized form. It stores:
- Fixed header (row-count, ncols, col-list-off) via deftype fields
- Column descriptor block referenced by col-list-off

Registration (same pattern as EveArray and Obj):
```clojure
(ser/register-cljs-to-sab-builder!
  (fn [v] (dataset? v))
  (fn [ds] ... build manifest, return IEveRoot ...))

(ser/register-header-constructor!
  EVE_DATASET_SLAB_TYPE_ID
  (fn [_sab blk-off] ... read manifest, reconstruct dataset ...))

(ser/register-header-disposer!
  EVE_DATASET_SLAB_TYPE_ID
  (fn [blk-off] ... free manifest + col descriptor block ...))
```

### Atom integration

```clojure
(def a (eve/mmap-atom "/tmp/data"
         (dataset {:price (arr/eve-array :float64 [10.5 20.3])})))

(swap! a (fn [ds]
           (add-column ds :doubled
             (functional/mul (column ds :price) 2.0))))

;; Structural sharing: :price column keeps its slab offset
;; Only new: :doubled column + new manifest
;; Epoch GC collects old manifest
```

## Step 2: eve.tensor

### NDBuffer type

Defined via `eve-slab-deftype`:

```clojure
(eve-slab-deftype TensorMeta
  [^:int32 rank            ;; number of dimensions (1–8)
   ^:int32 dtype           ;; element type code
   ^:int32 data-off        ;; slab offset of backing EveArray
   ^:int32 elem-offset     ;; element offset into the array
   ^:int32 s0 ^:int32 d0   ;; stride/dim for axis 0
   ^:int32 s1 ^:int32 d1   ;; stride/dim for axis 1
   ^:int32 s2 ^:int32 d2   ;; stride/dim for axis 2
   ^:int32 s3 ^:int32 d3]) ;; stride/dim for axis 3
```

Fixed 4 dimensions covers most use cases (scalar, vector, matrix, 3D tensor).
For rank > 4, use a referenced descriptor block (same pattern as dataset
column list). Fits in a 128-byte slab block.

Element `[i0, i1, ..., in]` → flat index: `elem-offset + i0*s0 + i1*s1 + ...`

### Core API

```clojure
;; Construction
(from-array arr [3 4])              ;; wrap 12-elem EveArray as 3×4 matrix
(zeros :float64 [3 4])             ;; allocate + fill
(ones :int32 [2 3 4])              ;; allocate + fill

;; Shape ops (zero-copy — same backing EveArray)
(reshape t [4 3])                   ;; new shape, same data
(transpose t)                       ;; reverse strides
(transpose t [1 0 2])               ;; permute axes
(slice t 0 1)                       ;; select row 1 → rank-1 view

;; Element access
(mget t 1 2)                        ;; get element at [1,2]
(mset! t 1 2 42.0)                  ;; set element at [1,2]

;; Bulk
(emap + t1 t2)                      ;; element-wise → new array
(ereduce + 0 t)                     ;; reduce all elements
(shape t)                           ;; → [3 4]
(dtype t)                           ;; → :float64
(contiguous? t)                     ;; true if C-order strides

;; Materialization
(to-array t)                        ;; flatten to EveArray
(to-dataset t [:a :b :c :d])       ;; 2D → Dataset
```

### Slab serialization

Type-id: `EVE_TENSOR_SLAB_TYPE_ID = 0x20`

The TensorMeta slab block IS the serialized form. The backing EveArray
is a separate slab block referenced by `data-off`.

### Composition with Dataset

```clojure
(def prices (column ds :price))
(def mat (tensor/from-array prices [30 24]))
(def hourly (tensor/transpose mat))            ;; zero-copy
(def hourly-sum (tensor/reduce-axis + hourly 1))
(swap! a #(add-column % :hourly-sum (tensor/to-array hourly-sum)))
```

## Step 3: eve.dataset.functional + eve.dataset.argops

### eve.dataset.functional (.cljc)

Element-wise operations on EveArrays. Works on standalone arrays AND
dataset columns (columns ARE EveArrays).

```clojure
;; Arithmetic (return new EveArray)
(add col1 col2)
(sub col1 col2)
(mul col1 col2)            ;; or (mul col 2.0) for scalar
(div col1 col2)

;; Aggregations (return scalar)
(sum col)                  ;; delegates to SIMD for int32
(mean col)
(min-val col)
(max-val col)

;; Comparison (return EveArray :uint8 of 0/1)
(gt col 15.0)
(lt col1 col2)
(eq col val)

;; Mapping
(emap f col)               ;; (f elem) → new array
(emap2 f col1 col2)        ;; (f a b) → new array
```

### eve.dataset.argops (.cljc)

Index-space operations. Return EveArray :int32 of indices.

```clojure
(argsort col)              ;; indices that would sort
(argsort col :desc)
(argfilter col pred)       ;; indices where (pred elem) true
(argmin col)               ;; index of minimum (scalar)
(argmax col)               ;; index of maximum (scalar)
(arggroup col)             ;; {value → EveArray of indices}
(take-indices col idx-arr) ;; gather: new array from indices
```

Compose with dataset:
```clojure
(let [idx (argops/argsort (column ds :price) :desc)]
  (dataset/reindex ds idx))
```

## Future: Foreign Format Adapters

Arrow, Parquet, FlatBuffers, Protobuf, etc. are **not** native Eve types.
They are adapters that convert to/from Eve's native types (EveArray, Dataset).

The serializer registration system (`register-cljs-to-sab-builder!`,
`register-header-constructor!`, `register-header-disposer!`) is the general
extensibility point. A future `eve.arrow` namespace would:

1. Read Arrow IPC → construct EveArrays (zero-copy for numeric: same bytes)
2. Write EveArrays → Arrow IPC (zero-copy for numeric)
3. Optionally define Arrow-specific slab types (validity bitmaps, offset
   arrays for strings) via `eve-slab-deftype` + serializer registration

Users who need Arrow null semantics, string columns, or other Arrow-specific
features define those types using the same macro and registration pattern.
Eve is the extensible value encoding — the tools are there, the default
format stays simple.

10. **Run new tests** — verify new functionality

## Constraints

- No changes to `atom.cljc` or `coalesc.cljc`
- No changes to existing serialization logic
- CLJS codegen path in macro is untouched (only additive JVM path)
- All column data lives in existing EveArray slab blocks (type-id 0x1D)
- All I/O through ISlabIO (cross-platform by construction)
- Native column format = EveArray. Arrow = future adapter.
