# Async Utilities

`cljs-thread` exports several utilities for async coordination from `cljs-thread.core`: `on-when`, `on-watch`, and `sleep`. These are primarily used for bootstrapping — waiting for conditions that can't be checked synchronously, such as worker readiness or atom initialization.

## `sleep`

Block the current worker thread for a given number of milliseconds:

```clojure
(require '[cljs-thread.core :as t])

(t/sleep 1000)  ;; blocks for 1 second
```

`sleep` uses `Atomics.wait` on worker threads. On the screen thread, it's a no-op (the screen thread must never block). In the Service Worker fallback path, it uses the SW's sleep/wake protocol.

## `on-when` — Poll Until Condition

`on-when` polls a condition at regular intervals and executes a body when the condition becomes truthy. Returns a Promise.

```clojure
(require '[cljs-thread.core :as t :refer [on-when]])

;; Wait for an element to appear in the DOM
(on-when (.querySelector js/document "#ready")
  (println "Element is ready!"))

;; With options
(on-when (some? @my-resource)
  {:duration 50       ;; poll every 50ms (default: 2ms)
   :max-time 5000}    ;; timeout after 5 seconds (default: 10000ms)
  (println "Resource loaded:" @my-resource))
```

### Options

| Key | Default | Description |
|-----|---------|-------------|
| `:duration` | `2` | Poll interval in milliseconds |
| `:max-time` | `10000` | Timeout in milliseconds |
| `:resolve` | `nil` | Post-process function `(fn [result] ...)` |
| `:timeout-resolve` | `nil` | Called on timeout instead of rejecting |
| `:timeout-data` | auto | Context data included in timeout error (ns, line) |

If the condition is already truthy when `on-when` is called, the body executes immediately (via `Promise.resolve`).

On timeout, the returned Promise rejects with an error message containing the namespace, line number, and condition unless `:timeout-resolve` is provided.

## `on-watch` — Watch Atom Until Predicate

`on-watch` adds a watch to an atom and executes a body when the predicate on the atom's value becomes true. The watch is automatically removed after firing.

```clojure
(require '[cljs-thread.core :as t :refer [on-watch]])

(def ready? (atom false))

;; Execute body when atom becomes true
(on-watch ready? true?
  (println "System is ready!"))

;; Trigger from somewhere else
(reset! ready? true)
;; prints: System is ready!
```

### Options

```clojure
(on-watch my-atom some?
  {:max-time 30000}    ;; timeout after 30 seconds (default: 30000ms)
  (println "Atom has a value"))
```

| Key | Default | Description |
|-----|---------|-------------|
| `:max-time` | `30000` | Timeout in milliseconds |
| `:resolve` | `nil` | Post-process function `(fn [result] ...)` |
| `:wkey` | auto | Custom watch key (auto-generated by default) |
| `:timeout-data` | auto | Context data included in timeout error |

If the predicate is already satisfied when `on-watch` is called, the body executes immediately.

## When to Use These

These are bootstrapping utilities — useful when initialization ordering isn't deterministic:

```clojure
;; Wait for a peer worker to connect before sending messages
(on-watch peers #(contains? % :my-worker)
  (msg/post :my-worker {:dispatch :init}))

;; Wait for IndexedDB to be ready
(on-watch db-open? true?
  (idb-get :saved-state yield))
```

For application-level reactive patterns, prefer standard `add-watch` on shared atoms, or use Reagent/re-frame subscriptions with the DOM proxy.
